Этот файл содержит краткую выжимку лекционного материала (сети, порты, шлюзы, Cloudflare Tunnels), сценарий 5-минутного выступления для защиты проекта, а также ответы на вероятные вопросы преподавателя.

## ЧАСТЬ 1: Краткий конспект лекций (Сети и Деплой)

### 1. Основы сетевого взаимодействия
*   **IP-адрес (IPv4):** Уникальный числовой идентификатор устройства в сети (например, `192.168.1.5` — локальный, `8.8.8.8` — публичный).
*   **Публичные vs Приватные IP:** 
    *   *Приватные (локальные):* Используются внутри домашних или корпоративных сетей (диапазоны `192.168.x.x`, `10.x.x.x`, `172.16.x.x` - `172.31.x.x`). Они не видны из интернета.
    *   *Публичные:* Уникальны во всем интернете. Нужны для того, чтобы сервер был доступен извне.
*   **Шлюз (Gateway):** Устройство (обычно роутер), которое соединяет локальную сеть (LAN) с глобальной (WAN/Интернет). Все пакеты, направленные во внешнюю сеть, идут через шлюз.
*   **NAT (Network Address Translation):** Технология на шлюзе, которая позволяет множеству устройств с локальными IP выходить в интернет под одним общим публичным IP-адресом.
*   **Порты (Ports):** Числовые идентификаторы (от 0 до 65535) для конкретных программ/служб на одном устройстве. Позволяют разделять сетевой трафик.
    *   `22` — SSH (удаленное управление)
    *   `80` — HTTP (незащищенный веб-трафик)
    *   `443` — HTTPS (защищенный веб-трафик)
    *   `5432` — PostgreSQL (база данных)
    *   `3000` / `5000` — Стандартные порты для веб-приложений и API.

### 2. DNS (Domain Name System)
Система, переводящая понятные человеку доменные имена (например, `google.com`) в машинные IP-адреса.
*   **Иерархическая структура:** Запросы идут от корневых серверов (Root) к зонам верхнего уровня (Top-Level Domains, TLD: `.com`, `.dk`, `.tech`), а затем к авторитетным серверам имен конкретного домена.
*   **Основные типы записей:**
    *   **A-запись (Address):** Связывает доменное имя напрямую с IPv4-адресом сервера.
    *   **CNAME-запись (Canonical Name):** Псевдоним. Указывает, что один домен ссылается на другое доменное имя (например, `api.mydomain.com` ведет на `my-tunnel.cloudflare.com`).
    *   **MX-запись (Mail Exchange):** Указывает серверы, отвечающие за прием почты для домена.

### 3. Безопасность и Firewalls (Межсетевые экраны)
*   **Firewall (Брандмауэр):** Фильтрует входящий и исходящий трафик на основе правил безопасности.
*   **Уровни фильтрации (по модели OSI):**
    *   *Простые (L3/L4):* Фильтруют трафик только по IP-адресам и портам (например, открыть порт 80, закрыть все остальные).
    *   *Next Generation Firewall (NGFW / L7):* Анализируют содержимое пакетов (Deep Packet Inspection), распознают конкретные приложения и защищают от известных уязвимостей (IDS/IPS).
*   **Проблема прямого экспонирования:** Открывать порты базы данных (`5432`) или SSH (`22`) напрямую наружу на публичном IP небезопасно, так как боты постоянно сканируют сеть и пытаются подобрать пароли.

### 4. Cloudflare Tunnels (Безопасный деплой без публичного IP)
Вместо открытия портов на файрволе и покупки дорогого публичного IP-адреса используется технология туннелирования:
1.  На сервере Ubuntu запускается легкий демон `cloudflared`.
2.  Он сам устанавливает **исходящее** защищенное соединение (туннель) с серверами Cloudflare.
3.  Cloudflare направляет веб-трафик пользователей через этот туннель прямо на локальный порт нашего приложения (например, `5000`).
4.  **Плюсы:**
    *   Публичный IP сервера скрыт.
    *   Не нужно открывать входящие порты в брандмауэре сервера.
    *   Бесплатная защита от DDoS-атак и WAF (Web Application Firewall) от Cloudflare «из коробки».

## ЧАСТЬ 2: Сценарий презентации на 5 минут

*Совет по таймингу: говорите уверенно, не торопитесь, укладывайтесь в рамки слайдов.*

### Слайд 1: Введение и Архитектура Проекта (1 минута)
> *"Приветствую! Мой проект — это **Movie Pulse Bot**. Это сервис, который собирает тренды кино (из TMDB, Reddit и YouTube), анализирует обсуждения с помощью нейросети Gemini и публикует ежедневные обзоры в Telegram-каналы на разных языках.*
> *Архитектура состоит из трех основных частей: базы данных PostgreSQL, Python-скрипта (коллектора/бота) и веб-API на FastAPI для отдачи статистики. Для деплоя я выбрал современный и безопасный стек: Docker Compose, сервер под управлением Ubuntu и технологию Cloudflare Tunnel."*

### Слайд 2: Подготовка и перенос на сервер (1 минута)
> *"Сначала я арендовал виртуальный сервер (VPS) на Ubuntu. Для управления сервером я подключался по протоколу **SSH** командой:*
> `ssh oleh@ip_адрес_сервера`
> *Для переноса файлов конфигурации я использовал Git. Я настроил репозиторий на GitHub, сделал коммит и на сервере просто стянул код командой `git pull`.*
> *Конфиденциальные ключи и пароли мы вынесли в файл `.env`, который добавлен в `.gitignore` и хранится только локально на сервере, предотвращая утечку секретов в репозиторий."*

### Слайд 3: Докеризация и Docker Compose (1.5 минуты)
> *"Для обеспечения независимости от окружения мы упаковали API и бота в **Docker-контейнеры**.*
> *В Dockerfile мы применили лучшие практики безопасности: настроили запуск приложения от непривилегированного пользователя `appuser` вместо `root`.*
> *Для оркестрации всех сервисов используется **Docker Compose**. Мы описали три службы в файле `docker-compose.yml`:*
> 1.  `postgres` — база данных PostgreSQL (использует Docker-volume для сохранения данных на диске).
> 2.  `api` — FastAPI-сервис, отдающий данные.
> 3.  `cf-tunnel` — клиент Cloudflare Tunnel.
>
> *Основные команды для работы с Docker на сервере:*
> *   `docker compose up -d --build` — собирает образы (если были изменения в коде) и запускает контейнеры в фоновом режиме (`-d`).
> *   `docker compose down` — останавливает и удаляет запущенные контейнеры.
> *   `docker compose ps` — показывает статус запущенных контейнеров и их здоровье (healthcheck).
> *   `docker compose logs -f api` — выводит логи приложения в реальном времени."*

### Слайд 4: Сетевая безопасность и Cloudflare Tunnel (1 минута)
> *"Главная фишка нашего деплоя — это сетевая безопасность. Мы не стали покупать публичный IP для сервера и открывать порты наружу. Вместо этого мы запустили контейнер `cf-tunnel` с токеном от Cloudflare.*
> *Он создает исходящее соединение, связывая доменное имя, настроенное в DNS Cloudflare (через CNAME-запись), с локальным портом `5000` внутри нашей Docker-сети. Таким образом, наше API защищено от внешнего сканирования портов и DDoS-атак."*

### Слайд 5: Заключение (30 секунд)
> *"В результате проект успешно развернут. API работает стабильно (статус healthy), база данных защищена, а все секреты и пароли находятся в безопасности. Проект готов к масштабированию. Спасибо за внимание, готов ответить на ваши вопросы!"*

## ЧАСТЬ 3: Возможные вопросы преподавателя и ответы на них

#### Впрос 1: Зачем вы используете Docker и Docker Compose, если можно было просто запустить Python-скрипт на сервере?
*   **Ответ:** *"Docker решает проблему «у меня работает, а на сервере нет» (dependency hell). Он изолирует приложение со всеми его зависимостями и версией Python в легком контейнере. Docker Compose позволяет одной командой (`docker compose up`) поднять всю инфраструктуру (базу данных PostgreSQL, API и туннель) с настроенной внутренней сетью, лимитами и правильным порядком запуска."*

#### Вопрос 2: Как устроена безопасность вашего API? Что мешает любому пользователю забрать ваши данные?
*   **Ответ:** *"Доступ к API защищен авторизацией по API-ключу через HTTP-заголовок `X-API-Key`. В коде реализована строгая валидация: если переменная `API_KEY` в окружении сервера не задана или пуста, доступ блокируется для всех. Кроме того, в FastAPI настроен Rate Limiter (`slowapi`), ограничивающий частоту запросов до 100 в минуту с одного IP, что защищает API от парсинга и перегрузки."*

#### Вопрос 3: Почему вы не открыли порт 5432 (Postgres) наружу, чтобы подключаться к базе из дома через DBeaver/pgAdmin?
*   **Ответ:** *"Открытие портов БД наружу — серьезная уязвимость. Злоумышленники постоянно сканируют порт 5432. Наша база данных Postgres работает во внутренней изолированной сети Docker-compose и доступна только для контейнера с API. Если мне нужно будет подключиться к базе напрямую, я сделаю это безопасно через SSH-туннель."*

#### Вопрос 4: Как Cloudflare Tunnel узнает, куда именно на сервере направлять трафик?
*   **Ответ:** *"В контейнер передается токен туннеля (`CF_TUNNEL_TOKEN`). Клиент `cloudflared` подключается к серверам Cloudflare и сообщает им о готовности принимать трафик. В панели управления Cloudflare настроено правило маршрутизации: трафик, приходящий на наш домен, перенаправляется на локальный адрес `http://movie-pulse-api:5000` внутри Docker-сети."*

#### Вопрос 5: Что делает строчка `useradd -m appuser && USER appuser` в вашем Dockerfile?
*   **Ответ:** *"По умолчанию процессы в Docker-контейнере запускаются под пользователем `root`. Если злоумышленник сможет эксплуатировать какую-то уязвимость в коде веб-сервера (например, RCE), он получит права суперпользователя внутри контейнера и потенциально сможет совершить побег на хост-систему. Запуск от имени ограниченного пользователя `appuser` сводит эту угрозу к минимуму."*
